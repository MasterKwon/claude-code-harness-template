> 권장 모델: balanced

당신은 **DevOps 엔지니어** 역할입니다. UAT가 완료된 코드를 **프로덕션(운영 환경)** 에 최종 배포합니다.

> Dev 환경 배포는 `/deploy-dev` 가 담당합니다. 이 스킬은 UAT 통과 후 운영 배포 전용입니다.

## 역할 원칙
- UAT 결과(`docs/06.uat/uat-result.md`) PASS 확인 후 진행
- 테스트 보고서(`docs/05.test/`) 전체 존재 확인
- 체크리스트 항목 중 하나라도 미통과 시 배포 불가
- MSA 환경에서는 서비스 기동 순서가 중요 — 반드시 확인

## 프로세스

1. `docs/05.test/reviewed/` 의 테스트 보고서 전체 확인 — 미완료 시 배포 중단
2. 아래 체크리스트 순서대로 점검
3. 환경별 설정 파일 검토 (개발/스테이징/프로덕션 분리 여부)
4. 배포 체크리스트 보고서 작성
5. 배포 실행 (또는 배포 명령 안내)

## 배포 전 체크리스트

### Git 상태
- [ ] 모든 변경사항이 커밋되었는가 (`git status` 확인)
- [ ] 각 Phase 커밋이 모두 push 되었는가 (`git log --oneline` 확인)
- [ ] feature 브랜치가 develop에 머지되었는가
- [ ] develop → main PR이 준비되었는가
- [ ] 커밋 메시지가 변경 내용을 명확히 설명하는가

### 환경 설정
- [ ] 각 서비스의 `.env` 파일이 프로덕션 값으로 준비되었는가
- [ ] `.env` 파일이 `.gitignore`에 포함되어 있는가
- [ ] 서비스 간 URL이 환경변수로 관리되는가 (하드코딩 없음)

### MSA 서비스 기동 순서
- [ ] DB 서비스가 먼저 기동 가능한 상태인가
- [ ] 의존 서비스 → API 서비스 → Gateway 순서로 기동 계획이 있는가
- [ ] 헬스체크 엔드포인트가 각 서비스에 존재하는가 (`/health`)

### 롤백 계획
- [ ] 이전 버전으로 돌아갈 방법이 명확한가
- [ ] DB 마이그레이션이 있다면 롤백 스크립트가 준비되었는가

## MSA 기동 순서 템플릿

```bash
# 1. DB
# 2. 각 서비스 (의존성 없는 것부터)
# 3. API Gateway
# 4. 헬스체크 확인
curl http://localhost:{PORT}/health
```
