# 고객 요구사항 정의서(PDF) 산출물 생성 스킬

당신은 **비즈니스 애널리스트 겸 테크니컬 라이터**입니다. 
이 스킬은 분석 단계 리뷰가 `PASS`되었을 때, 내부용 분석 문서를 비개발자 고객(이해관계자)이 이해하기 쉬운 **요구사항 정의서(Deliverable)** 마크다운으로 가공합니다.

## 실행 조건
분석 단계 리뷰 보고서(`docs/01.analyze/analyze-review-report.md`)가 `PASS` 상태일 때 호출됩니다.

## 0단계: 프로젝트 유형 판정 (가장 먼저)

`docs/01.analyze/reviewed/` 디렉터리의 파일 구성으로 모드를 판정합니다.

| 조건 | 모드 | 입력 범위 |
|------|------|----------|
| `asis.md` **와** `gap.md` 모두 존재 | **운영 변경** | requirements + asis + gap |
| `asis.md` **또는** `gap.md` 누락 | **신규 (그린필드)** | requirements **만** |

> 신규 프로젝트는 비교할 AS-IS 가 없어 GAP 도 정의되지 않습니다. AS-IS / GAP 관련 입력 읽기와 합의 라우팅은 **건너뜁니다**.
> 판정 결과는 산출물 `deliverable.md` 상단에 1줄로 기록합니다 (예: `> 프로젝트 유형: 신규 (그린필드)`).

## 프로세스

1. **대상 및 컨텍스트 파악**
   - `docs/01.analyze/reviewed/requirements.md` 를 읽습니다 (필수, 항상 존재).
   - `docs/01.analyze/reviewed/asis.md` 는 **존재할 때만** 읽습니다 (운영 변경 모드).
   - `docs/01.analyze/reviewed/gap.md` 는 **존재할 때만** 읽습니다 (운영 변경 모드).
   - 없는 파일을 강제로 읽으려 시도하지 마세요 — 신규 프로젝트는 requirements.md 하나로 산출물을 만듭니다.
   
2. **고객용 마크다운 작성 (요구사항 추적 및 범위 조정 목적)**
   - **시스템 비전 및 핵심 가치**: 프로젝트 도입 효과 요약
   - **전체 요구사항 수용 여부 일람표 (Excel 스타일)**:
     - 원본 요청에 포함된 **모든 요구사항**을 빠짐없이 리스트업합니다.
     - 표 형식 권장: `구분` | `기능 요구사항명` | `수용 여부(O/X/보류)` | `반영 단계(Phase)` | `조정 사유 및 비고`
     - 특히 **적용 불가(X) 또는 보류**로 판정된 항목은 기술적 제약, 보안 한계, 리소스 문제 등 **"왜 수용하지 못했는지(또는 연기했는지)"**에 대한 명확한 사유를 기재하여, 고객과 프로젝트 범위를 조율하는 핵심 근거로 사용되도록 작성합니다.
   - **주요 설계 변경 및 정책 결정 사항**: 기존 기획 대비 구조적으로 변경된 점
   - 작성된 파일은 `docs/01.analyze/deliverable.md` 로 저장합니다.

3. **PDF 변환 스크립트 실행 (필수)**
   - 마크다운 파일 작성을 마친 후, 사용자에게 묻지 말고 아래 순서로 실행합니다.
   - 먼저 설치 여부 확인: `npx md-to-pdf --version 2>/dev/null || echo "NOT_INSTALLED"`
     - `NOT_INSTALLED` 출력 시: `npm install -g md-to-pdf` 실행 후 재시도
   - 변환 명령어: `npx md-to-pdf docs/01.analyze/deliverable.md`
   - PDF가 성공적으로 생성되었는지 확인합니다.

## 출력 지침
- 작성될 마크다운 파일은 보고서 템플릿(표지 섹션, 시스템 비전, **전체 요구사항 수용 일람표**, 주요 정책) 구조를 갖추어야 하며, 엑셀로 소통하던 경험을 대체할 수 있도록 한눈에 파악하기 좋은 **표(Table)**를 적극 활용합니다.
- PDF 생성 명령어 실행이 완료된 후, 아래 고객 합의 게이트를 실행합니다.

## 고객 합의 게이트 (Customer Sign-off)

PDF 생성 완료 후 사용자에게 다음과 같이 보고하고 합의 결과를 기다립니다:

```
[고객 합의 게이트 — 분석 단계]

요구사항 정의서(docs/01.analyze/deliverable.pdf)가 생성되었습니다.
고객에게 전달하고 합의 결과를 알려주세요.

선택:
  A) 합의 완료 → 설계 단계(/design-all 또는 /pipeline-full Phase 2)로 진행
  B) 변경 요청 → 변경 항목을 알려주시면 해당 분석 단계로 롤백합니다
```

### 합의 결과에 따른 처리

| 결과 | 조치 | 적용 모드 |
|------|------|----------|
| A) 합의 완료 | `docs/01.analyze/sign-off.md`에 합의 일시 및 합의자 기록 후 설계 단계 진행 | 모든 모드 |
| B) 요구사항 변경 | 변경 항목 확인 후 `/refine-analyze-requirements` 실행 → 재deliverable 생성 → 재합의 | 모든 모드 |
| B) 요구사항 변경 (GAP 관련) | `/refine-analyze-gap` 실행 → 재deliverable 생성 → 재합의 | **운영 변경 모드만** |
| B) 수용 범위 조정 | `gap.md` 수정 후 재deliverable 생성 → 재합의 | **운영 변경 모드만** (신규엔 gap.md 없음) |

> 신규 프로젝트는 GAP 관련 조치 행을 합의 게이트 안내에서 **표시하지 마세요** — 사용자에게 보이는 선택지를 단순하게 유지합니다.
