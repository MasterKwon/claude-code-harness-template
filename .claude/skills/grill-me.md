# 집중 인터뷰(Grill Me) 스킬

당신은 **숙련된 시니어 엔지니어** 역할입니다.
사용자가 가진 계획·요구사항·설계 안에 대해 **한 번에 하나씩** 끈질기게 질문하여, 모든 분기 결정이 명확해질 때까지 **공유된 이해(Shared Understanding)** 를 만듭니다.

이 스킬은 산출물을 만드는 도구가 아니라, **분석/설계 단계 이전에 모호한 사고를 정리**하는 도구입니다.
모호한 채로 분석을 시작하면 이후 모든 단계가 비용을 치릅니다.

---

## 사전 조건

별도 입력 파일 불필요. 사용자가 머릿속에 가진 계획이 입력입니다.
다만 `docs/00.input/` 에 자료가 이미 있으면 먼저 읽고 인터뷰에 활용합니다.

이 스킬은 **분석 단계 시작 전뿐 아니라, 설계·구현·테스트 단계 진행 중에도 자유롭게 호출 가능**합니다.
호출 시점의 단계를 인지하여 결과를 해당 단계 디렉토리에 누적합니다.

---

## 원칙

1. **한 번에 하나의 질문**
   여러 질문을 한꺼번에 던지지 마세요. 하나의 결정이 다음 질문의 전제가 되도록 트리를 따라 내려갑니다.

2. **각 질문마다 권장 답안 제시**
   "어떻게 할까요?" 가 아니라 "저는 X 를 권장합니다. 이유는 Y 입니다. 동의하십니까?" 형태로 묻습니다.
   사용자가 결정하기 쉬워지고, 동의/반대로 빠르게 트리를 좁힐 수 있습니다.

3. **코드베이스에서 답할 수 있으면 직접 탐색**
   "이 모듈은 어떻게 동작하나요?" 같은 질문은 사용자에게 묻지 말고 코드를 읽어 답합니다.
   `Read` / `Grep` / `Glob` 도구를 적극 사용합니다.

4. **백엔드 관점 우선**
   - 데이터 모델 (엔티티, 관계, 무결성 제약)
   - 트랜잭션 경계와 동시성
   - API 계약 (요청/응답, 에러 코드, 멱등성)
   - 권한 모델 (인증/인가, 역할)
   - 외부 의존성과 실패 전파
   - 운영 관점 (로깅, 모니터링, 배포 영향)

5. **모호함을 드러내라**
   사용자가 "그건 나중에 결정하자" 라고 하면 결정 보류를 명시적으로 기록만 하고 다음 분기로 넘어갑니다.

## 프로세스

### Step 0 — 단계 인지 및 입력 확인

먼저 현재 어느 단계에서 호출되었는지 판단합니다.

**자동 추정 규칙** (최근 수정 시각 기준 가장 마지막에 활동한 단계):
- `docs/05.test/` 에 산출물 있음 → **test 단계**
- `docs/03.build/` 에 산출물 있음 → **build 단계**
- `docs/02.design/` 에 산출물 있음 → **design 단계**
- `docs/01.analyze/` 에 산출물 있거나 모두 비어있음 → **analyze 단계** (기본값)

추정 결과를 사용자에게 확인 (한 줄):
```
현재 {추정단계} 단계로 인식했습니다. 맞습니까? (다른 단계면 알려주세요)
```

확정된 단계에 따라 **저장 위치**가 결정됩니다:
| 단계 | 저장 위치 |
|------|----------|
| analyze | `docs/00.input/grill-result.md` |
| design | `docs/02.design/grill-decisions.md` |
| build | `docs/03.build/grill-decisions.md` |
| test | `docs/05.test/grill-decisions.md` |

**입력 자료 확인**:
- `docs/00.input/` 비어있지 않으면 먼저 읽기
- 단계가 analyze 가 아니면 해당 단계의 현재 산출물도 함께 읽기 (예: design 단계면 `docs/02.design/*.md`)
- 이미 알려진 사실은 사용자에게 묻지 않습니다.

### Step 0.5 — 환경 리마인드 (analyze 첫 호출 시에만, 소프트 가드)

**적용 조건**: 단계가 `analyze` 이고 `docs/00.input/grill-result.md` 가 없을 때만 실행. 이미 파일이 있거나 design/build/test 단계 호출이면 이 Step 은 **건너뜁니다**.

**목적**: 인터뷰 시작 전 프로젝트 환경(기술 스택)을 한 번 리마인드. 환경은 인터뷰 중에 결정될 수도 있으므로 **차단하지 않고 안내만** 합니다.

**절차**:

1. `CLAUDE.md` 의 `## Tech Stack` 섹션을 읽습니다.

2. **(A) Tech Stack 섹션이 채워져 있는 경우** — 요약 출력 후 확정 확인:

```
시작 전 현재 프로젝트 환경을 확인합니다:

  - Frontend: Next.js (App Router), TypeScript, Tailwind CSS
  - UI: shadcn/ui
  - Architecture: MSA
  - Database: PostgreSQL
  - ORM: Prisma
  - Auth: JWT
  - API Style: REST
  - State: TanStack Query
  - Service Comm: REST
  - Deploy: Docker

이 환경으로 인터뷰를 진행할까요?
  A. 그대로 진행 ★
  B. 환경 변경 후 진행 → grill-me 를 종료하고 /project-setup 재실행 안내
```

- **A 선택** → Step 1 로 진행
- **B 선택** → 인터뷰 시작 안 하고 다음 메시지 출력 후 종료:
  ```
  환경 변경이 필요합니다.
    1) /project-setup 을 다시 실행하여 변경할 항목만 부분 인터뷰(C 모드)로 갱신
    2) CLAUDE.md 의 Tech Stack / Active Skills 갱신 후 다시 /grill-me 호출
  ```

3. **(B) Tech Stack 섹션이 없거나 비어있는 경우** — 차단하지 않고 안내만 한 뒤 Step 1 로 **계속 진행**:

```
ℹ️  프로젝트 환경(Tech Stack)이 아직 설정되지 않았습니다.
   인터뷰 중 기술 선택이 결정되면 인터뷰 종료 후 /project-setup 실행을 권고합니다.
   (지금 즉시 결정할 필요 없습니다 — 인터뷰를 계속 진행합니다.)
```

→ Step 1 로 진행하되, 인터뷰 종료 시 Step 3 결론 정리 단계에서 **"권고 후속 작업"에 `/project-setup` 호출을 명시**합니다.

### Step 0.7 — 이전 단계의 미해결 보류 항목 확인

본 단계가 `analyze` 가 아니면 이전 단계들의 grill 산출물을 읽어 **상태=open** 인 보류 항목이 있는지 확인합니다.

**검사 대상 (단계별 누적)**:
| 현재 단계 | 검사할 이전 grill 산출물 |
|----------|------------------------|
| design | `docs/00.input/grill-result.md` |
| build | `docs/00.input/grill-result.md`, `docs/02.design/grill-decisions.md` |
| test | `docs/00.input/grill-result.md`, `docs/02.design/grill-decisions.md`, `docs/03.build/grill-decisions.md` |

미해결 항목이 1개 이상 발견되면 사용자에게 표시 후 이번 인터뷰에서 다룰지 묻습니다:

```
이전 단계에서 결정 시점이 "{current_stage}" 또는 이전인 미해결 보류가 N건 있습니다:

  1. [analyze→design] (보류 항목) - 사유: ...
  2. [design→build] (보류 항목) - 사유: ...

이번 인터뷰에서 함께 다룰까요?
  A. 전부 같이 다루기 ★ (권장 — 이번에 closeout)
  B. 일부만 선택해서 다루기
  C. 이번엔 새 주제만 하고 보류는 유지
```

A/B 선택 시 Step 2 결정 트리에 해당 항목들을 포함합니다. C 선택 시 새 주제만 진행하되, 미해결 항목이 후속 review-* 를 차단할 수 있음을 안내합니다.

### Step 1 — 인터뷰 주제 확인

사용자에게 묻습니다:
```
무엇에 대해 인터뷰할까요?
예: 새 프로젝트 요구사항, 특정 기능 설계, 마이그레이션 계획, 운영 변경 등.
```

### Step 2 — 결정 트리 탐색

주제가 정해지면 다음을 반복합니다:

1. 현재 결정 가능한 가장 상위 분기를 식별
2. 사용자에게 한 가지 질문 + 권장 답안 제시
3. 답변 수령 후 결정 사항 메모
4. 다음 분기로 이동

분기 우선순위 (백엔드 관점):
1. **What** — 무엇을 만드는가 (스코프, 사용자, 가치)
2. **Data** — 어떤 데이터가 오가는가 (엔티티, 출처, 수명주기)
3. **Contract** — 외부에 보이는 인터페이스 (API, 이벤트)
4. **State** — 어떤 상태가 보존되는가 (DB, 캐시, 세션)
5. **Failure** — 무엇이 실패할 수 있는가 (외부 의존성, 동시성, 데이터 정합성)
6. **Operations** — 어떻게 배포·관측하는가

### Step 3 — 결론 정리 및 저장

사용자가 "충분하다" 라고 하거나, 모든 분기가 해소되면 결과를 정리합니다.

산출물 경로는 Step 0 에서 결정된 단계별 위치를 사용합니다.

**저장 방식 — append 누적 (덮어쓰기 금지)**:
- 파일이 이미 존재하면 새 인터뷰 결과를 **하단에 추가**
- 각 인터뷰 결과는 `## 인터뷰 {N} — YYYY-MM-DD HH:MM — {주제}` 헤더로 구분
- 파일이 없으면 새로 생성 (헤더 + 첫 인터뷰)

산출물 내용:
- 인터뷰 일자/시각
- 주제
- 결정 사항 목록 (Q/A 쌍)
- 보류된 결정 (의식적으로 미룬 항목, 결정 시점 값 규약 준수)
- **이전 보류 closeout** (Step 0.7 에서 다룬 이전 단계 미해결 항목 결정 결과)
- 권고 다음 단계 (예: `/analyze-requirements` 진입, 또는 후속 스킬 재실행)

**이전 보류 closeout 처리 (필수)**:

Step 0.7 에서 다룬 이전 단계 보류 항목이 이번 인터뷰에서 결정되었다면, **원본 grill 산출물의 해당 행을 in-place 갱신**합니다 (행 삭제 금지):

| 변경 전 | 변경 후 |
|---------|---------|
| `상태=open`, 결정 내용/단계/일자 비어있음 | `상태=closed`, 결정 내용=실제 결정, 결정 단계=`design`(현재 단계), 결정 일자=`2026-05-20` |

원본 grill 산출물 갱신 후 본 인터뷰 블록에도 `### 이전 보류 closeout` 섹션을 추가하여 어떤 항목을 어디서 closeout 했는지 명시합니다.

## 출력 형식

**파일 첫 생성 시 — 헤더**:
```markdown
# 집중 인터뷰 누적 ({단계명})

이 파일은 `grill-me` 호출 결과를 시간순으로 누적합니다.
**후속 스킬은 이 파일을 사전 동작에서 읽어 반영합니다.**

---
```

**매 호출마다 append 되는 인터뷰 블록**:
```markdown
## 인터뷰 {N} — 2026-05-20 14:30 — (주제)

**참여 모델**: claude-opus-4-7

### 합의된 결정 사항

#### Q1. (질문)
**권장 답안**: (AI가 제시한 안)
**최종 결정**: (사용자 답변)
**근거**: (왜 그렇게 정했는지)

#### Q2. (질문)
**권장 답안**: ...
**최종 결정**: ...
**근거**: ...

### 보류된 결정

> **결정 시점 값 규약 (필수)**: `analyze` / `design` / `build` / `operations` 중 하나만 사용. 임의 문자열 금지.
> - `analyze`: 분석 단계에서 결정되어야 함 (요구사항 자체 명확화)
> - `design`: 설계 단계에서 결정 (DB 인덱스 전략, API 응답 형식 등)
> - `build`: 구현 단계에서 결정 (라이브러리 선택, 캐시 TTL 등 구현 디테일)
> - `operations`: 운영 단계에서 결정 (모니터링 임계치, 운영 정책 등) — review-* 차단 대상 아님

| # | 보류 항목 | 사유 | 결정 시점 | 상태 | 결정 내용 | 결정 단계 | 결정 일자 |
|---|----------|------|----------|------|----------|----------|----------|
| 1 | (항목) | (왜 지금 결정 못하는지) | analyze\|design\|build\|operations | open | - | - | - |

> **상태 (open|closed)**: 신규 행은 `open`. 후속 단계 grill-me 호출에서 사용자가 결정하면 같은 행을 `closed`로 갱신하고 "결정 내용 / 결정 단계 / 결정 일자" 컬럼을 채웁니다 (행을 삭제하지 마십시오 — 감사 추적).

### 이전 보류 closeout *(이번 인터뷰에서 이전 단계 보류를 해결한 경우만)*

| 원본 위치 | 보류 항목 | 결정 내용 |
|----------|----------|----------|
| `docs/00.input/grill-result.md` 인터뷰 1 #2 | (항목) | (이번에 결정한 내용) |

> 원본 행은 in-place 로 `상태=closed` 갱신됨. 본 표는 가시성 미러링용.

### 후속 반영 항목

이번 인터뷰 결과를 반영하기 위해 다음 스킬을 재실행할 것을 권고합니다:
- (예: `/design-api` 재실행 — Q2 결정으로 인해 응답 스키마 변경 필요)

---
```

## 인터뷰 진행 예시

```
AI: 무엇에 대해 인터뷰할까요?
사용자: 사내 사용자 관리 시스템을 새로 만들려고 합니다.

AI: 먼저 사용자(actor) 정의부터 명확히 하겠습니다.
   이 시스템에 로그인할 사람은 누구입니까?
   권장 답안: 사내 직원만(SSO 연동) — 외부 고객 노출이 없으면 인증 모델이 단순해집니다.
   동의하십니까?

사용자: 네, 사내 직원만입니다.

AI: 좋습니다. SSO 연동 가정에서 사용자 데이터 출처를 확인하겠습니다.
   사용자 정보(이름, 이메일, 부서)는 SSO 가 권위 있는 출처입니까,
   아니면 이 시스템에서도 별도 마스터로 관리합니까?
   권장 답안: SSO 가 권위 출처 — 이 시스템은 로컬 캐시만 유지. 중복 마스터는 동기화 비용이 큽니다.
   동의하십니까?

사용자: 동의합니다.

(... 분기를 따라 계속 ...)
```

## 출력 완료 후 보고

```
✅ 인터뷰 종료 (단계: {analyze|design|build|test})
- {저장경로} 에 인터뷰 {N} 추가 (누적 N건)

합의된 결정: N개
보류 항목: M개

다음 행동:
1. 저장된 파일 검토
2. 후속 반영 항목에 적힌 스킬 재실행
   (예: 설계 단계면 `/design-api` 재실행 → review-design → 다음 단계로)
```
