# 작업 단위 집중 인터뷰 (Grill Task) 스킬

당신은 **익숙한 직원** 역할입니다.
운영 시스템에서 변경 작업 1건을 시작하기 전, `docs/context/` 의 도메인 노트를 먼저 읽고
**이미 아는 사실은 패스, 모르는 것만 사용자에게 묻는** 방식으로 작업 맥락을 좁힙니다.

이 스킬은 산출물 작성용이 아니라 **작업 진입 시점의 맥락 정렬과 권장 Phase 흐름 결정**을 위한 도구입니다.
`grill-me` 가 큰 주제(프로젝트/마이그레이션) 인터뷰라면, `grill-task` 는 단일 작업 인터뷰입니다.

---

## 사전 조건

`docs/context/INDEX.md` 가 존재해야 합니다.
없으면 다음 메시지를 출력하고 종료합니다:

```
docs/context/INDEX.md 가 없습니다. 사용 시점에 따라 다음 중 하나를 따르세요.

▶ 운영 시스템에서 변경 작업을 시작하려는 경우:
    /maintenance-init 을 먼저 한 번 실행하세요. CONTEXT 스켈레톤이 생성됩니다.

▶ 신규 개발 중인 경우:
    /grill-task 는 운영 누적 노트가 있어야 동작합니다.
    큰 주제 인터뷰는 /grill-me 를 사용하세요.
    운영 배포 후 /maintenance-init 을 실행한 시점부터 /grill-task 가 활성화됩니다.
```

`docs/00.input/` 에 자료가 있으면 함께 읽습니다.

---

## Step 0 — 작업 요청 수신

사용자에게 작업 요청을 받습니다:

```
작업 요청: (예: 환불 사유 enum 추가)
변경 유형(선택): 오류수정 / 기능개선 / 내부개선
```

작업 slug 를 결정합니다 (영문 소문자 + 하이픈, 예: `add-refund-reason-enum`).

---

## Step 1 — CONTEXT INDEX 매칭

`docs/context/INDEX.md` 를 읽고 작업 키워드를 도메인 한 줄 요약과 비교하여 도메인을 매칭합니다.

후보가 1개 → 바로 다음 Step.
후보가 2~3개 → 사용자에게 확인:

```
매칭 도메인 후보:
  1순위: order — 주문 생성/취소/환불 라이프사이클
  2순위: payment — PG 연동, 결제수단 관리

→ 어느 도메인으로 진행할까요? (복수 선택 가능)
```

**매칭 실패 시**: 사용자에게 묻습니다:
```
기존 도메인에 매칭되는 항목이 없습니다.
1) 신규 도메인을 만든다 (도메인명: ?)
2) 기존 도메인 X 에 흡수한다
→ 선택?
```
신규 도메인 선택 시 `docs/context/{domain}.md` 스켈레톤 (B 6섹션 빈 헤더)을 생성하고 INDEX 에 한 줄 추가합니다.

---

## Step 2 — 도메인 노트 로드 및 "아는 사실" 확인

매칭된 도메인 노트를 전체 로드한 후, **이번 작업과 관련 있는 항목**을 추려 사용자에게 보여줍니다:

```
이 작업과 관련해서 제가 이미 알고 있는 사실:
  ✓ (개요) order 는 결제를 payment 도메인에 위임
  ✓ (함정) Order.status 의 'PARTIAL_REFUND' 는 임시값, 2026-Q3 정리 예정
  ✓ (결정 이력 #4) 환불 사유는 DB enum 대신 string + Zod 검증 (2026-03-12)
  ✓ (변경 패턴) 환불 사유 추가는 분기마다 1~2회 발생

→ 이 사실 중 더 이상 맞지 않거나 수정이 필요한 항목이 있나요?
```

사용자 응답:
- 모두 맞음 → Step 3
- 일부 수정/삭제 → 변경 내용 메모. Step 4 산출물의 "확인된 사실" 섹션에 기록 (Phase 11 의 CONTEXT 갱신 단계에서 후보로 자동 활용)

**CONTEXT 충실도 판정**:
- **풍부**: 결정 이력 ≥ 5건 AND 함정 ≥ 3건
- **보통**: 결정 이력 1~4건 OR 함정 1~2건
- **희박**: 결정 이력 0건 AND 함정 0건 (스켈레톤 그대로인 경우 포함)

이 값은 Step 4 의 권장 Phase 흐름 산출에 사용됩니다.

---

## Step 3 — CONTEXT 에 없는 것만 인터뷰

원칙:
1. **한 번에 하나의 질문** + 권장 답안 동봉 ("저는 X 를 권장합니다. 이유는 Y. 동의하십니까?")
2. 코드에서 답할 수 있는 것은 코드를 읽어 답함 (`Read`/`Grep`/`Glob`)
3. CONTEXT 에 이미 있는 내용은 묻지 않음
4. 모호한 결정은 보류로 명시 기록 후 다음 분기로 이동

분기 우선순위 (백엔드 관점):
1. **What** — 이번에 변경할 정확한 범위
2. **Data** — 데이터 형태/제약 변경 여부
3. **Contract** — API/이벤트의 외부 약속 변경 여부
4. **Failure** — 동시성/정합성/실패 시나리오
5. **Operations** — 배포/관측에 미치는 영향

질문이 더 나오지 않거나 사용자가 "충분하다" 라고 하면 종료.

**"물어볼 게 없음" 판정**: CONTEXT 가 풍부 + 작업 범위가 사소(텍스트 수정·enum 추가 등)인 경우 Step 3 을 건너뛰고 Step 4 로 직행할 수 있습니다.

---

## Step 4 — 결과 저장 + 권장 Phase 흐름 제안

산출물: `docs/00.input/grill-task-{YYYYMMDD}-{slug}.md`

```markdown
# Grill Task — {작업 제목}

**일자**: 2026-05-18
**Slug**: add-refund-reason-enum
**매칭 도메인**: order (1순위), payment (2순위)
**CONTEXT 충실도**: 풍부 / 보통 / 희박
**변경 유형**: 오류수정 / 기능개선 / 내부개선

---

## 작업 요청
(원문)

## 확인된 사실 (CONTEXT 기반)
- ✓ (수정 없음) ... 
- ✏️ (수정 필요) 기존: ... → 신규: ...
- ➕ (신규 발견) ...

## 합의된 결정 사항

### Q1. ...
**권장 답안**: ...
**최종 결정**: ...
**근거**: ...

(반복)
```

---

## 안전망 (절대 생략 불가 Phase)

CONTEXT 가 아무리 풍부해도 다음은 반드시 진행합니다:

| Phase | 사유 |
|-------|------|
| 6.5 impact-check | 계획 외 변경/잘못 들어간 코드를 잡는 마지막 자동 게이트 |
| 7 review-all | 코드 자체 품질·보안 — 도메인 익숙도와 무관 |
| 8 변경 기능 테스트 + 회귀 High | 회귀 High 는 노트로 대체 불가 |
| 8.5 deploy-dev | UAT 전 안전 검증 |
| 9 UAT | 사람의 최종 확인 |
| 10 deploy-prd | 배포 자체는 생략 불가 |
| 11 변경이력 + CONTEXT 갱신 | 다음 작업을 위한 학습 단계 — 빠지면 시스템이 정체됨 |

**생략/축약 가능**: Phase 2, 3, 4, 5, 5.5, 8 회귀 Med/Low
