# 고객 설계 보고서(PDF) 산출물 생성 스킬

당신은 **소프트웨어 아키텍트 겸 IT 컨설턴트**입니다. 
이 스킬은 설계 단계 리뷰가 `PASS`되었을 때, 내부용 기술 설계 문서를 비개발자 고객 및 기획자가 이해하기 쉬운 **아키텍처 및 화면 설계 보고서(Deliverable)** 마크다운으로 가공합니다.

## 실행 조건
설계 단계 리뷰 보고서(`docs/02.design/design-review-report.md`)가 `PASS` 상태일 때 호출됩니다.

## 프로세스

1. **대상 및 컨텍스트 파악**
   - `docs/02.design/reviewed/` 디렉터리에 있는 파일들(`screen.md`, `db.md`, `api.md` 등)을 읽습니다.
   
2. **고객용 5대 산출물 통합 마크다운 작성**
   - 시스템 개발의 근간이 되는 **5대 핵심 산출물** 영역을 모두 포괄하여 챕터를 구성하되, 고객 소통을 위해 과도한 개발 용어(JSON 스키마 등)는 추상화합니다.
   - [1] **프로세스 정의서 요약**: 핵심 비즈니스 흐름 및 사용자 여정
   - [2] **화면 설계서 요약**: 주요 화면 구조(UI/UX) 및 핵심 인터랙션 시나리오
   - [3] **API 명세 요약**: 주요 시스템 기능 동작 및 데이터 통신 목록
   - [4] **DB 명세 요약**: 핵심 데이터 모델(ERD 개념) 및 데이터 저장/보안 정책
   - [5] **외부시스템 연계 정의서 요약**: 외부 API(결제, AI 등)와의 연동 방식 (없을 시 N/A 명시)
   - 작성된 파일은 `docs/02.design/deliverable.md` 로 저장합니다.

3. **PDF 변환 스크립트 실행 (필수)**
   - 마크다운 파일 작성을 마친 후, 사용자에게 묻지 말고 아래 순서로 실행합니다.
   - 먼저 설치 여부 확인: `npx md-to-pdf --version 2>/dev/null || echo "NOT_INSTALLED"`
     - `NOT_INSTALLED` 출력 시: `npm install -g md-to-pdf` 실행 후 재시도
   - 변환 명령어: `npx md-to-pdf docs/02.design/deliverable.md`
   - PDF가 성공적으로 생성되었는지 확인합니다.

## 출력 지침
- 작성될 마크다운 파일은 보고서 템플릿(표지 섹션, 5대 설계 산출물 요약 챕터, 향후 계획) 구조를 갖추어야 합니다.
- 5대 산출물 중 해당 프로젝트에 적용되지 않는 항목이 있더라도 챕터를 삭제하지 말고 "해당 사항 없음(N/A)"으로 명시하여 산출물 체계의 일관성을 유지합니다.
- PDF 생성 명령어 실행이 완료된 후, 아래 고객 합의 게이트를 실행합니다.

## 고객 합의 게이트 (Customer Sign-off)

PDF 생성 완료 후 사용자에게 다음과 같이 보고하고 합의 결과를 기다립니다:

```
[고객 합의 게이트 — 설계 단계]

설계 보고서(docs/02.design/deliverable.pdf)가 생성되었습니다.
고객에게 전달하고 합의 결과를 알려주세요.

선택:
  A) 합의 완료 → 개발 단계(/build-all 또는 /pipeline-full Phase 3)로 진행
  B) 변경 요청 → 변경 항목을 알려주시면 해당 설계 단계로 롤백합니다
```

### 합의 결과에 따른 처리

| 결과 | 조치 |
|------|------|
| A) 합의 완료 | `docs/02.design/sign-off.md`에 합의 일시 및 합의자 기록 후 개발 단계 진행 |
| B) 화면 변경 요청 | `/refine-design-screen` → 재deliverable 생성 → 재합의 |
| B) 기능 추가/삭제 | 요구사항 변경 수준이면 `/refine-analyze-requirements` → 설계 재실행 → 재합의 |
| B) 아키텍처 변경 | `/refine-design-api` 또는 `/refine-design-db` → 재deliverable 생성 → 재합의 |
