# 변경 영향도 검사

구현된 코드 변경이 기존 기능에 미치는 영향을 분석하여 리뷰와 회귀 테스트 범위를 결정합니다.

---

## 입력

- `docs/01.analyze/reviewed/gap.md` — 계획된 변경 범위
- `git diff` — 실제 변경된 코드 (base branch 기준)

---

## 실행 단계

### Step 1 — 실제 변경 파일 목록 추출

```bash
git diff --name-only main...HEAD
```

변경 파일을 레이어별로 분류합니다:

| 레이어 | 해당 경로 패턴 |
|--------|-------------|
| DB | `prisma/schema.prisma`, `prisma/migrations/**` |
| API | `app/api/**/route.ts`, `lib/**`, `schemas/**` |
| Screen | `app/**/page.tsx`, `components/**` |
| 공통 | `store/**`, `hooks/**`, `utils/**` |

### Step 2 — 의존성 추적

각 변경 항목이 기존 코드에 미치는 영향을 아래 전파 방향으로 추적합니다:

```
DB 컬럼/테이블 변경
  → 해당 컬럼을 사용하는 API route.ts
    → 해당 API를 호출하는 Screen 컴포넌트

API 엔드포인트/응답 구조 변경
  → 해당 API를 호출하는 Screen 컴포넌트
  → 해당 API에 의존하는 TanStack Query hook

공통 컴포넌트 변경
  → 해당 컴포넌트를 import하는 모든 Screen

Store/Hook 변경
  → 해당 store/hook을 사용하는 모든 컴포넌트
```

### Step 3 — 범위 교차 검증

GAP에서 계획한 변경 범위와 실제 변경된 파일을 비교합니다:

- **계획에 없는 파일 변경** → 즉시 보고, 원인 확인 필요
- **계획된 변경이 누락** → 구현 미완료 항목 확인
- **일치** → 정상 진행

### Step 4 — 영향도 분류

영향받는 각 기존 기능에 대해 영향도를 분류합니다:

| 영향도 | 기준 |
|--------|------|
| **High** | 인증 흐름, 핵심 데이터 구조(PK/FK), 다수 기능이 의존하는 공통 로직 |
| **Medium** | API 응답 구조, 공유 컴포넌트, Store 인터페이스 |
| **Low** | UI 스타일, 비필수 기능, 독립적인 유틸리티 |

### Step 5 — 회귀 테스트 목록 생성

영향받는 기존 기능별로 회귀 테스트 시나리오를 생성합니다.

---

## 출력

`docs/03.build/impact-check.md`에 저장합니다.

```markdown
# 변경 영향도 검사 결과

> 검사 일시: YYYY-MM-DD
> 기준 브랜치: main

## 범위 교차 검증

| 구분 | 내용 |
|------|------|
| GAP 계획 파일 수 | N개 |
| 실제 변경 파일 수 | N개 |
| 계획 외 변경 | 없음 / (파일명 목록) |
| 구현 누락 | 없음 / (항목 목록) |

## 영향도 분석

| 변경 항목 | 레이어 | 영향받는 기존 기능 | 영향도 | 회귀 테스트 시나리오 |
|---------|--------|----------------|------|------------------|
| | | | | |

## 회귀 테스트 대상 요약

### High
- (항목)

### Medium
- (항목)

### Low
- (항목)

## 다음 단계 권고

- High 항목 존재: 사용자 확인 후 진행
- High 항목 없음: review-all 진행
```

---

## 다음 단계

### 계획 외 변경 발견
즉시 작업을 중단하고 사용자에게 보고합니다. 해당 변경이 의도된 것인지 확인 후:
- 의도된 변경 → GAP에 추가 후 계속
- 의도치 않은 변경 → 해당 파일 롤백 후 계속

### High 영향도 항목 존재
사용자에게 보고하고 진행 여부 확인을 받습니다.
확인 후 `review-all`에서 해당 항목을 중점 검토합니다.

### 모든 항목 Medium 이하
`docs/04.review/report.md` 생성을 위해 `review-all`로 진행합니다.
