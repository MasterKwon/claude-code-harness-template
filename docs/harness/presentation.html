01

배경과 문제 의식

AI 코딩 도구를 쓰기 시작하면 처음에는 빠릅니다.
그런데 시간이 지나면…

AI 코딩의 현실

😤  "이거 지난번에도 물어봤는데 또 잊어버렸네"

😰  "AI가 만들어준 코드인데 왜 DB와 API가 따로 놀지?"

😩  "어디서부터 다시 시작해야 하지?"

😱  "테스트는 했는데 배포하고 나서 다른 게 깨졌어"

핵심 문제  —  AI는 강력하지만, 가이드가 없으면 매 세션마다 처음부터 다시 설명해야 합니다.

단계별 수동 진행단계 누락, 산출물 불일치
AI에게 전부 맡기기범위 초과, 예상 못한 변경

02

하네스 엔지니어링이란

AI의 강력한 능력을 통제된 방향으로 흐르게 만드는 구조물

개념

"실수가 발생하면 사과하는 대신,
다시는 같은 실수가 반복되지 않도록 환경을 개선한다."

— Andrej Karpathy & Jeff Hashimoto

하네스 (Harness)

말에 채우는 마구(馬具).
힘을 방향 있게 쓰도록 제어하는 장치.

→ AI의 능력을 통제된 방향으로 흐르게 만드는 구조물

적용 원칙 3가지

  • Surgical Changes — 요청 범위만 건드린다
  • Think Before Coding — 분석 → 설계 → 구현 순서
  • Simplicity First — 가장 단순한 코드

하네스 구성 요소

Claude Code 하네스
│
├── CLAUDE.md          프로젝트 컨텍스트
│                      기술스택 · 가이드라인 · 원칙
│
├── .claude/commands/  슬래시 커맨드 진입점
│                      49개
│
├── .claude/skills/    단계별 실행 로직
│                      상세 지침 · 품질 게이트
│
├── .claude/stacks/    기술스택별 패턴
│                      플러그인 방식으로 교체 가능
│
└── .claude/hooks/     자동화 트리거
                       수정 후 타입체크 등

CLAUDE.md

세션이 바뀌어도 프로젝트 컨텍스트가 유지되는 영구 메모리

commands/ + skills/

슬래시 명령어로 시작 → 스킬 파일이 단계별 지침 실행

stacks/

Prisma ↔ TypeORM처럼 기술스택 파일 한 줄로 전환

hooks/

파일 수정 즉시 TypeScript 타입체크 자동 실행

파이프라인 3개 체계

상황에 맞는 파이프라인을 선택합니다

파이프라인 3개 체계 — 한눈에

/pipeline-full

신규 프로젝트

처음부터 새로 만들기 · 분석 → 설계 → 구현 → 리뷰 → 테스트 → UAT → 배포

/pipeline-change

기능 추가·변경

기존 프로젝트 · AS-IS 먼저, GAP이 작업 범위 결정 · 회귀 테스트 포함

/pipeline-maintenance

운영 유지보수

고객 변경 요청 반복 처리 · Patch 모드 구현 · 6종 요청 유형 분류

/pipeline-full — 신규 프로젝트

[분석]                     [설계]           [설계리뷰]  [TC설계]       [구현]
analyze-requirements  →   design-screen  →           →  design-tc →  build-db
analyze-asis          →   design-db      → review-   →             →  build-api
analyze-gap           →   design-api     →  design   →             →  build-screen
                                                              ↓
[영향도 검사]          [리뷰]          [AI QA · Local]      [Dev 배포]    [UAT]     [운영 배포]
impact-check       →  review-all  →  test-db          →  deploy-dev →  UAT    →  ship
                                  →  test-api
                                  →  test-screen, test-e2e
Phase명령산출물
0/project-setup프로젝트 초기화, CLAUDE.md
1/analyze-allrequirements.md, asis.md, gap.md
2/design-all + /review-design + /design-tc설계 3종 + UAT 체크리스트
3/build-all + /impact-check소스코드 + 영향도 보고서
4~5/review-all → /test-all리뷰·테스트 보고서
5.5/deploy-devDev 환경 배포 → UAT 수행
7/ship운영 배포 체크리스트

/pipeline-change — 기능 추가·변경

핵심 차이 — AS-IS 분석이 먼저, GAP이 작업 범위를 결정합니다.
GAP에 없는 코드는 건드리지 않습니다.

[현황 파악]           [변경 정의]               [범위 확정]
analyze-asis      →  analyze-requirements  →   analyze-gap
(기존 코드 먼저)      (변경할 것만)                (작업 목록 확정)

      ↓
[영향 범위만 설계]    [영향 범위만 구현]       [리뷰 + 테스트]         [배포]
GAP 결과 기반        GAP 결과 기반            회귀 테스트 포함    →   deploy-dev → ship

pipeline-full 과의 차이

  • AS-IS 분석이 먼저 실행
  • GAP 항목만 설계·구현
  • 회귀 테스트 별도 실행

GAP 분류 기준

  • 신규 — 없던 것을 추가
  • 변경 — 기존 것을 수정
  • 유지 — 건드리지 않음
  • 제거 — 삭제 대상

/pipeline-maintenance — 운영 유지보수

[요청 분류]           [설계 변경]          [영향도 검증]
customer-request →   설계 문서 수정    →  impact-check
6종 유형 분류         (변경 범위만)

      ↓
[구현 · Patch 모드]   [리뷰]         [AI QA]       [배포 · 선택]
기존 코드 최소 수정 →  review-all →  test-*     →  deploy-dev → ship

↺  다음 변경 요청이 오면 Phase 1부터 반복

코드설명시작 단계

SCR화면 레이아웃·UX 변경화면 설계 수정

실제 동작 방식

품질 게이트 · 롤백 맵 · 스택 플러그인

품질 게이트 & 롤백 맵

품질 게이트 — 각 Phase 완료 시 자동 검증

✅요구사항: Must/Should/Could 우선순위 부여됨
✅DB 설계: ERD + 인덱스 전략 포함
✅API 설계: Swagger 명세 포함
✅구현: TypeScript 타입 에러 없음 (hook 자동 체크)
✅리뷰: High 심각도 없음
✅테스트: E2E 전체 통과

⛔ 게이트 실패 시 다음 Phase로 진행하지 않습니다.

롤백 맵 — 문제 위치마다 명시된 복구 경로

발견 위치         원인           조치
──────────────────────────────────────────
테스트 실패   → API 구현 오류  →
  /refine-build-api → 리뷰 → 테스트 재실행

UAT FAIL      → 기능 누락     →
  /refine-design-* → 구현 → 테스트 → UAT

운영 배포 실패 → 환경설정 오류 →
  직접 수정 → /ship 재실행

01.WebDesktop 적용 사례

실제 프로젝트에 적용하며 발견한 13가지 취약점과 보완 과정

발견된 취약점 V1 ~ V7

#취약점보완 방법
V1분석 품질 게이트 부재각 분석 단계에 체크리스트 추가
V2설계 리뷰 단계 누락review-design 커맨드 + Phase 2.5 추가
V3구현 후 레이어 정합성 미검증impact-check + Phase 3.5 추가
V4유지보수 파이프라인 부재pipeline-maintenance 신규 구축
V5고객 요청 처리 체계 없음customer-request + 6종 분류 체계
V6Dev 배포 단계 누락deploy-dev 커맨드 추가
V7UAT와 AI 테스트 혼재AI QA(Local) / UAT(사람) 명확히 분리

발견된 취약점 V8 ~ V13

#취약점보완 방법
V8UAT 체크리스트 사후 작성design-tc를 구현 전(Phase 2.7)으로 이동
V9롤백 경로 미명시파이프라인 내 롤백 맵 전면 작성
V10요구사항 변경 시나리오 없음변경 발생 시점별 대응 절차 추가
V11refine-* 커맨드 일부 누락전체 주요 단계에 refine 커맨드 완비
V12설계 산출물 리뷰 미분리review-analyze / review-design 분리
V13유지보수 초기화 절차 없음maintenance-init 커맨드 추가

13개 취약점 보완 → 커맨드 40개에서 49개로 확장 · 파이프라인 2개 → 3개
