01

배경과 문제 의식

AI 코딩 도구를 쓰기 시작하면 처음에는 빠릅니다.
그런데 시간이 지나면…

AI 코딩의 현실

😤  "이거 지난번에도 물어봤는데 또 잊어버렸네"

😰  "AI가 만들어준 코드인데 왜 DB와 API가 따로 놀지?"

😩  "어디서부터 다시 시작해야 하지?"

😱  "테스트는 했는데 배포하고 나서 다른 게 깨졌어"

핵심 문제  —  AI는 강력하지만, 가이드가 없으면 매 세션마다 처음부터 다시 설명해야 합니다.

하네스 vs 나만의 방식 — 뭐가 달라요?

관점 나만의 방식 하네스 엔지니어링
컨텍스트 유지 매 세션마다 직접 설명 CLAUDE.md가 자동 로드
프롬프트 품질 사람마다, 날마다 다름 커맨드로 표준화 · 재현 가능
AI 범위 제어 사후 확인 (이미 건드린 뒤) GAP 사전 정의 → 범위 초과 차단
팀 온보딩 개인 노하우 → 공유 어려움 .claude/ 폴더 공유로 즉시 동기화
실수 반복 방지 기억에 의존 CLAUDE.md에 기록 → 영구 반영
💡 지금 방식이 잘 맞으면 바꿀 필요 없습니다. 이 구조가 더 필요해지는 순간은 — 팀이 커지거나, 프로젝트가 길어질 때입니다.
02

하네스 엔지니어링이란

AI의 강력한 능력을 통제된 방향으로 흐르게 만드는 구조물

개념

"실수가 발생하면 사과하는 대신,
다시는 같은 실수가 반복되지 않도록 환경을 개선한다."

— Andrej Karpathy & Jeff Hashimoto

하네스 (Harness)

말에 채우는 마구(馬具).
힘을 방향 있게 쓰도록 제어하는 장치.

→ AI의 능력을 통제된 방향으로 흐르게 만드는 구조물

적용 원칙 3가지

  • Surgical Changes — 요청 범위만 건드린다
  • Think Before Coding — 분석 → 설계 → 구현 순서
  • Simplicity First — 가장 단순한 코드

CLAUDE.md

세션이 바뀌어도 프로젝트 컨텍스트가 유지되는 영구 메모리

commands/ + skills/

슬래시 명령어로 시작 → 스킬 파일이 단계별 지침 실행

stacks/

Prisma ↔ TypeORM처럼 기술스택 파일 한 줄로 전환

hooks/

파일 수정 즉시 TypeScript 타입체크 자동 실행

자주 받는 현실 질문

🔧 CLAUDE.md 관리 비용이 더 크지 않나요?

처음 1회 세팅(약 30분)이 전부입니다. 이후에는 실수가 생길 때만 해당 규칙 한 줄을 추가합니다. 오히려 반복 설명이 사라져 세션당 시간이 줄어요.

⚠️ CLAUDE.md가 오래되면 오히려 독이 됩니다. 6개월에 한 번 리뷰를 권장합니다.

🛠️ 우리 기술스택과 안 맞으면요?

stacks/ 폴더의 파일 하나만 교체하면 됩니다. Prisma → TypeORM, REST → GraphQL 등 기술스택 교체 비용은 파일 1개 수정입니다. 없는 스택은 직접 추가할 수 있어요.

👥 팀원 여러 명이 동시에 쓰면요?

.claude/ 폴더를 Git으로 관리합니다. CLAUDE.md, commands/, stacks/ 모두 버전 관리 대상입니다. 팀 규칙 변경 시 PR로 리뷰 후 머지하면 전원 동기화됩니다.

🔄 Cursor / Copilot 쓰는데 굳이 바꿔야 하나요?

바꿀 필요 없습니다. CLAUDE.md · commands/ 구조는 사고방식입니다. 다른 도구에도 비슷하게 적용할 수 있어요. 단, slash command 자동 실행은 Claude Code 전용입니다.

🧑 단계가 늘었는데 학습 부담은 없나요?

v2.2.0에서 추가된 3단계(grill-me, design-prompt-gen, test-ui-chrome)는 모두 선택적 Bridge 단계입니다. 필요할 때만 실행하면 됩니다.

핵심 흐름(/pipeline-full, /pipeline-maintenance)은 변하지 않았어요. 처음엔 기존 흐름만 익히고, 모호한 요구사항·시각 검토·UX 확인이 필요해질 때 하나씩 도입하면 됩니다.

📉 도입 전후 차이가 체감되나요?

가장 먼저 느끼는 변화는 "같은 실수가 줄어든다"입니다. CLAUDE.md에 기록된 규칙은 다음 세션부터 AI가 자동으로 지킵니다. 반복 디버깅 시간이 줄어드는 게 1주일 내 체감됩니다.

WHY 정의 먼저 → AS-IS → GAP으로 작업 범위 확정 · 회귀 테스트 포함

파이프라인이 2개인 이유

/pipeline-full 신규 개발

백지 상태에서 시작할 때 사용합니다.

사용 예시

  • 새 프로젝트 처음 개발
  • 완전히 새로운 기능 모듈 추가
  • 레거시를 버리고 새로 재작성
요구사항 → 설계 → 구현 → QA → 배포
전체 SDLC를 처음부터 밟습니다.
/pipeline-maintenance 유지보수

이미 돌아가는 코드베이스를 바꿀 때 사용합니다.

사용 예시

  • 계산 오류, 잘못된 데이터 표시 수정
  • 조회 화면 추가, 필드 추가
  • 성능 개선, 리팩토링
AS-IS 먼저 파악 → GAP 범위 확정
영향받는 레이어만 수정합니다.

신규 → 백지에서 전체 설계   |   유지보수 → 현재 코드 먼저, 범위 최소화

단계별 티어 배분

동작 방식 · 역할 분담

품질 게이트 & 롤백 맵

품질 게이트 — 각 Phase 완료 시 자동 검증

✅요구사항: Must/Should/Could 우선순위 부여됨
✅DB 설계: ERD + 인덱스 전략 포함
✅API 설계: Swagger 명세 포함
✅구현: TypeScript 타입 에러 없음 (hook 자동 체크)
✅리뷰: High 심각도 없음
✅테스트: E2E 전체 통과

⛔ 게이트 실패 시 다음 Phase로 진행하지 않습니다.

롤백 맵 — 문제 위치마다 명시된 복구 경로

테스트 실패 ← API 구현 오류
/refine-build-api → 리뷰 → 테스트 재실행
UAT FAIL ← 기능 누락
/refine-design-* → 구현 → 테스트 → UAT
운영 배포 실패 ← 환경설정 오류
직접 수정 → /deploy-prd 재실행

AI가 삐끗했을 때 — 3가지 대응

1

범위를 벗어난 코드를 건드렸다

GAP에 없는 파일을 수정하거나 관련 없는 리팩토링을 시작하는 경우

Ctrl+C 즉시 중단 → git diff 로 범위 확인 → /refine-* 로 범위 재지정
2

계속 같은 방향으로 틀린다

여러 번 수정해도 같은 오류가 반복되는 경우 — 컨텍스트가 오염된 신호

/compact 컨텍스트 압축 → 설계서 파일을 다시 언급 → 커맨드 재실행
3

빌드가 깨졌다

타입 에러·런타임 오류로 실행이 안 되는 경우 — hooks가 먼저 잡아주기도 함

git stash 로 격리 → 에러 메시지를 그대로 붙여넣기 → /refine-build-* 재실행

공통 원칙: 멈추고 → 현재 상태 확인 → 범위를 좁혀서 재요청. 계속 밀어붙이면 복구가 더 어려워집니다.

자주 묻는 질문 — 처음 시작하는 분

Q. 슬래시 커맨드는 어디서 입력하나요?

Claude Code CLI 터미널 프롬프트에 /커맨드명 형태로 입력합니다. 채팅창이 아닌 터미널 인터페이스입니다.

Q. CLAUDE.md는 직접 써야 하나요?

/project-setup 커맨드를 실행하면 AI가 초안을 자동 생성합니다. 이후 팀 상황에 맞게 수정하면 됩니다.

Q. 처음 시작할 때 정확히 무엇을 하면 되나요?

템플릿 레포를 clone → .claude/ 폴더를 내 프로젝트에 복사 → Claude Code에서 /project-setup 실행. 끝입니다.

Q. AI가 범위를 벗어났는지 어떻게 알아요?

GAP 문서에 없는 파일이 git diff에 나타나거나, 요청하지 않은 리팩토링이 시작되면 신호입니다. Ctrl+C로 즉시 중단하세요.

Q. GAP 범위 승인, 기준이 뭔가요?

"이 파일들만 바꾸면 요구사항이 충족되는가?"가 기준입니다. 의심스러운 파일이 목록에 있으면 이유를 AI에게 먼저 물어보세요.

Q. 파이프라인 중간에 틀리면 처음부터 다시 하나요?

아닙니다. /refine-* 커맨드로 해당 단계만 재실행합니다. 앞 단계 산출물(docs/ 파일들)은 유지됩니다.

Q. 모델 티어는 단계마다 직접 바꿔야 하나요?

아닙니다. 커맨드 실행 시 자동으로 적용됩니다. CLAUDE.md에 티어를 한 번 정의해두면 이후 모든 커맨드가 자동으로 맞는 모델을 씁니다.

Q. UAT · SDLC · ERD · AS-IS · GAP이 뭔가요?

UAT — 사용자 인수 테스트 (실제로 잘 되는지 사람이 직접 확인)
SDLC — 소프트웨어 개발 전체 단계 (분석→설계→구현→테스트→배포)
ERD — DB 테이블 구조 다이어그램
AS-IS — 현재 코드 상태 분석 / GAP — 바꿔야 할 범위

💡 파이프라인이 자동으로 안내합니다 — 처음엔 5개 커맨드만 기억하세요

자주 묻는 질문 — 이미 AI를 쓰는 분

Q. 지금 방식이 잘 되는데 왜 바꿔야 하죠?

안 바꿔도 됩니다. 하지만 팀이 커지거나 프로젝트가 3개월을 넘어갈 때 구조 없이는 컨텍스트 유실과 품질 편차가 커집니다. 그 시점에 도입을 고려해보세요.

Q. CLAUDE.md가 오래되면 오히려 독 아닌가요?

맞습니다. 그래서 "실수할 때마다 한 줄 추가" 원칙을 지키고, 분기에 한 번 내용을 리뷰하는 습관이 중요합니다. 잘못된 CLAUDE.md는 없는 것보다 나쁠 수 있어요.

Q. 이 구조 자체를 다른 AI 도구에도 쓸 수 있나요?

사고방식(컨텍스트 문서화, 단계별 커맨드화, 범위 사전 정의)은 어느 도구에든 적용 가능합니다. slash command 자동화는 Claude Code 전용이지만 개념은 이식할 수 있어요.

Q. 팀 전체가 동시에 도입해야 하나요?

아닙니다. 혼자 먼저 써보고, 효과가 느껴지면 .claude/ 폴더를 팀 레포에 커밋하면 됩니다. 강제할 필요 없이 결과로 설득하는 방식을 권장합니다.

Q. AI가 GAP 범위를 무시하고 더 고치면요?

이게 핵심 문제입니다. GAP 문서에 "이 파일들만 수정 대상"이라고 명시하고, CLAUDE.md에 Surgical Changes 원칙을 적어두면 빈도가 줄어듭니다. 그래도 발생하면 Ctrl+C → /refine-* 재실행이 표준 대응입니다.

Q. 기존 레거시 프로젝트에 바로 적용할 수 있나요?

가능합니다. /pipeline-maintenance가 바로 그 용도입니다. AS-IS 분석부터 시작하기 때문에 코드베이스를 처음부터 AI에게 설명하지 않아도 됩니다.
