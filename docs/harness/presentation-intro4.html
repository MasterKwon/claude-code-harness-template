지금의 /pipeline-maintenance — 좁힌 것 vs 못 좁힌 것

지금의 /pipeline-maintenance 는 변경 코드를 GAP 범위로 좁힙니다. 그런데 정작 거치는 단계(분석·설계·리뷰·테스트)는 매 변경 작업마다 동일하게 풀 코스로 진행됩니다.

이미 좁힌 것 — 코드 변경 범위

  • GAP에 없는 코드는 건드리지 않음
  • 영향 레이어(Screen/API/DB)만 설계·구현
  • 전체 시스템 재작성이 아닌 변경 범위 한정

✓ 코드 차원의 surgical 변경은 이미 확보

아직 못 좁힌 것 — 거치는 단계 자체

Phase 2 · analyze-asis
Phase 3 · requirements
Phase 4 · analyze-gap
Phase 5 · design (영향 레이어)
Phase 6 · build
Phase 6.5~11 · 검증·배포·이력

한 줄 오타도, 익숙한 enum 추가도 — 분석부터 시작

결과: 우회가 발생한다

  • "이건 그냥 고쳐" 로 파이프라인 밖에서 처리
  • 변경 이력이 누락되고, 회귀 위험이 보이지 않는 곳에서 발생
  • 파이프라인을 만들었는데 정작 안 쓰는 역설
파이프라인이 현실과 안 맞으면 사용자가 우회합니다.
문제는 사용자가 아니라 파이프라인입니다.

신입 vs 1년 차 — 학습 곡선의 차이

사람은 일하면서 점점 빨라집니다. 차이는 노트가 쌓였기 때문입니다.

입사 첫 주 (신입)

  • 위키 보고 환경 세팅
  • "이건 어느 팀 일이에요?" 매번 질문
  • 한 작업당 1~2일
  • 주변 코드 다 들여다보고 변경 시작

1년 차 (익숙한 직원)

  • "근무유형 좀 봐줘" 한마디로 일 시작
  • 해당 도메인의 함정·관례를 머릿속에 보유
  • 한 작업당 1~2시간
  • 안전한 부분은 빠르게, 위험한 부분만 신중히
하네스의 운영 파이프라인도 사람처럼 학습해야 합니다.
매 작업을 신입처럼 굴리는 대신, 노트가 쌓이게 만듭시다.

빠졌던 것 — 직원의 머릿속 노트

실제 회사에는 세 가지 지식 저장소가 있습니다. 우리 하네스에는 그중 하나가 비어 있었습니다.

실제 회사우리 하네스특성
사내 위키CLAUDE.md정적 · 모두 공유 · 거의 변하지 않음
입사 첫 주 온보딩/maintenance-init1회성 · 기존 코드 역공학
직원 머릿속 / 노트없었음 ← 빠진 칸누적 · 작업할수록 풍부해짐

신설: docs/context/ 도메인 노트

docs/context/
  INDEX.md                  ← 도메인 목록 + 한 줄 요약
  work-management.md        ← 근무 관리 도메인 노트 (6섹션)
  payroll.md                ← 급여 도메인
  ...

6 섹션: 개요 · 코드 위치 · 외부 의존성 · 함정/주의사항 · 결정 이력 · 변경 패턴

/maintenance-init 한 번이면 메뉴·코드 모듈을 보고 빈 스켈레톤이 자동 생성됩니다. 채우는 건 작업하면서.
